# Database Plan

Status: conceptual schema only; not a migration  
Last reviewed: 2026-07-13

## 1. Conventions

- Primary keys: UUID, generated by the database unless offline creation requires a client UUID.
- User ownership: `user_id uuid not null references auth.users(id) on delete cascade`.
- Time: `timestamptz` in UTC; keep the user’s IANA time zone in settings.
- Standard timestamps: `created_at`, `updated_at`; add `deleted_at` only when recovery is a real requirement.
- Provider payloads: store normalized fields first and bounded `jsonb` only for data that cannot yet be normalized.
- Row Level Security (RLS): enabled before exposing a table through the Data API.
- Browser writes: allowed only for user-owned personal data and validated settings.
- Integration metadata and caches: browser-readable where useful, server-writable through Edge Functions.
- Token material: never stored in a browser-readable table.

The first migrations should define only Phase 1/2 tables. Integration tables can wait until the OAuth phase.

## 2. Core identity and settings

### `profiles`

**Purpose:** application profile corresponding one-to-one with a Supabase Auth user.

**Important columns**

- `id uuid primary key references auth.users(id)`
- `display_name text`
- `avatar_url text null`
- `locale text default 'en'`
- `time_zone text`
- `week_starts_on smallint`
- `created_at`, `updated_at`

**Relationships:** parent for user-facing identity; other tables should still reference `auth.users` directly through `user_id`.

**RLS:** authenticated user can select/update only `id = auth.uid()`. Insert through a trusted signup trigger or constrained self-insert. Public profile reads are not needed.

### `user_settings`

**Purpose:** durable preferences that should sync across devices.

**Important columns**

- `user_id uuid primary key`
- `theme text`
- `density text`
- `default_home_view text`
- `notification_preferences jsonb`
- `calendar_preferences jsonb`
- `privacy_preferences jsonb`
- `settings_version integer`
- `updated_at`

**Relationships:** one-to-one with `profiles`/Auth user.

**RLS:** owner select/insert/update only. Validate known keys and bounded JSON size. Sensitive credentials do not belong here.

## 3. Dashboard

### `dashboard_widgets`

**Purpose:** user-created widget instances and their settings.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `dashboard_key text default 'home'`
- `widget_type text`
- `title_override text null`
- `visible boolean default true`
- `settings jsonb default '{}'`
- `definition_version integer`
- `created_at`, `updated_at`

**Relationships:** layout items refer to `id` inside the versioned layout document. `widget_type` resolves through the client registry; it is not a database foreign key.

**RLS:** owner CRUD only. Validate allowed widget types, setting size, and settings schema in application/server code. Integration widgets still need a permission check at fetch time.

### `dashboard_layouts`

**Purpose:** atomic ordered layouts for each device class.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `dashboard_key text default 'home'`
- `device text check in ('desktop', 'mobile')`
- `version integer`
- `items jsonb`
- `created_at`, `updated_at`
- unique `(user_id, dashboard_key, device)`

**Relationships:** `items` contains widget instance IDs and grid geometry. Widget existence is validated before save.

**RLS:** owner CRUD only. A server/database check should reject foreign widget IDs, invalid dimensions, excessive item counts, and oversized JSON. Publish realtime changes only to the owner.

## 4. Connected services and synchronization

### `connected_accounts`

**Purpose:** safe metadata and health state for an external account; supports multiple accounts without requiring it in the first UI.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `provider text` (`google` initially)
- `provider_account_id text`
- `email text`
- `display_name text null`
- `avatar_url text null`
- `status text` (`active`, `reauth_required`, `error`, `disconnected`)
- `token_handle text` (opaque reference only; no token)
- `last_synced_at timestamptz null`
- `last_error_code text null`
- `last_error_at timestamptz null`
- `connected_at`, `disconnected_at`, `updated_at`
- unique `(user_id, provider, provider_account_id)`

**Relationships:** parent of permissions, caches, and sync jobs.

**RLS:** owner can select metadata. Connect, token-handle changes, provider identity changes, and disconnect completion are Edge Function/service-only. The browser may request an action but cannot write connection state directly.

Token ciphertext belongs in a private schema or external secret manager reachable only by trusted server code. It must not be exposed through Supabase Data API or Realtime.

### `integration_permissions`

**Purpose:** auditable record of requested and granted provider scopes by module/account.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `connected_account_id uuid`
- `module text`
- `scope text`
- `status text` (`granted`, `revoked`, `missing`)
- `granted_at`, `revoked_at`
- unique `(connected_account_id, module, scope)`

**Relationships:** many permissions belong to one connected account.

**RLS:** owner select only; server manages grants/revocations after checking the provider response. A user must not self-assert a granted scope.

### `integration_cache`

**Purpose:** bounded provider-response cache shared by widgets and module pages.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `connected_account_id uuid`
- `provider text`
- `resource_type text`
- `cache_key text`
- `payload jsonb`
- `etag text null`
- `source_updated_at timestamptz null`
- `fetched_at timestamptz`
- `expires_at timestamptz`
- `sensitivity text`
- unique `(connected_account_id, resource_type, cache_key)`

**Relationships:** account-scoped cached resources; detailed provider data can later move to dedicated tables if queries require it.

**RLS:** owner select only when the cache is intended for client display; Edge Function/service writes only. Apply short retention to Gmail content, purge on disconnect, and cap payload size. Never cache provider tokens.

### `sync_jobs`

**Purpose:** idempotent synchronization queue and operational history.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `connected_account_id uuid null`
- `job_type text`
- `status text` (`queued`, `running`, `succeeded`, `failed`, `cancelled`)
- `dedupe_key text`
- `cursor jsonb null`
- `attempt integer`
- `scheduled_for`, `started_at`, `finished_at`
- `error_code text null`, `error_summary text null`
- `created_at`
- partial unique index preventing duplicate active `dedupe_key`

**Relationships:** optionally belongs to one connected account; can produce cache rows and notifications.

**RLS:** owner can select sanitized status. Server creates/claims/updates jobs. Client-triggered refresh calls an Edge Function, which enforces cooldowns before creating a job.

## 5. Notifications

### `notifications`

**Purpose:** persistent in-app notification inbox for personal reminders, sync health, and actionable alerts.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `type text`
- `title text`
- `body text`
- `severity text`
- `entity_type text null`, `entity_id uuid null`
- `action_route text null`
- `read_at timestamptz null`
- `dismissed_at timestamptz null`
- `dedupe_key text null`
- `created_at`, `expires_at null`

**Relationships:** soft reference to a domain entity; do not require a polymorphic foreign key.

**RLS:** owner select and update read/dismiss fields. Domain notifications are created by trusted database functions or Edge Functions. Realtime is owner-scoped.

## 6. Personal productivity data

### `projects`

**Purpose:** outcomes that group tasks, notes, and events.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `name text`
- `description text null`
- `status text` (`planned`, `active`, `paused`, `completed`, `archived`)
- `start_date date null`, `target_date date null`
- `color_token text null` (approved design token name, not arbitrary CSS)
- `progress_mode text` (`manual`, `tasks`)
- `manual_progress smallint null`
- `created_at`, `updated_at`, `archived_at null`

**Relationships:** optional parent of tasks, notes, and calendar events.

**RLS:** owner CRUD. Child ownership policies must verify the project belongs to the same user.

### `tasks`

**Purpose:** actionable personal work.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `project_id uuid null`
- `parent_task_id uuid null`
- `title text`
- `description text null`
- `status text` (`inbox`, `todo`, `doing`, `done`, `cancelled`)
- `priority smallint`
- `due_at timestamptz null`
- `scheduled_start`, `scheduled_end` null
- `completed_at timestamptz null`
- `position numeric`
- `created_at`, `updated_at`

**Relationships:** belongs optionally to a project and parent task. A later tag join table can be added when task tags are required.

**RLS:** owner CRUD; project and parent references must point to owner records. Realtime owner-scoped. Index active lists by `(user_id, status, due_at)` and project views by `(user_id, project_id, status)`.

### `notes`

**Purpose:** durable personal knowledge and captured text.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `project_id uuid null`
- `title text`
- `content text`
- `content_format text default 'markdown'`
- `pinned boolean`
- `archived_at timestamptz null`
- `created_at`, `updated_at`
- generated/search column when full-text search is implemented

**Relationships:** optional project relationship. Add note tags/backlinks with dedicated join tables only when those features are scheduled.

**RLS:** owner CRUD and owner-only search. Realtime invalidates lists/details but must not overwrite unsaved editor content. Index user/update order and full-text search.

### `calendar_events`

**Purpose:** personal events and reminders; independent from official Khmer holidays.

**Important columns**

- `id uuid primary key`
- `user_id uuid`
- `project_id uuid null`
- `title text`
- `description text null`
- `starts_at`, `ends_at timestamptz`
- `all_day boolean`
- `time_zone text`
- `recurrence_rule text null`
- `recurrence_exceptions jsonb null`
- `reminder_minutes integer[]`
- `location text null`
- `created_at`, `updated_at`

**Relationships:** optional project; reminders can create notification rows.

**RLS:** owner CRUD. Validate start/end ordering, recurrence size, and owner project. Realtime owner-scoped and queried by bounded date range.

## 7. Khmer reference data

### `khmer_holidays`

**Purpose:** versioned official Cambodian public holidays and verified festival dates by year. This is not the lunar calculation engine and does not contain personal events.

**Important columns**

- `id uuid primary key`
- `year integer`
- `date date`
- `end_date date null`
- `name_km text`
- `name_en text`
- `category text` (`public_holiday`, `religious`, `traditional_festival`)
- `official boolean`
- `source_name text`
- `source_url text`
- `source_document_number text null`
- `published_at date null`
- `verified_at timestamptz null`
- `verification_status text` (`draft`, `verified`, `superseded`)
- `version integer`
- unique `(year, date, name_km, version)`

**Relationships:** shared reference data; calendar views combine it at query/display time with user-owned `calendar_events`.

**RLS:** authenticated users can select only verified rows. Writes are admin/service-only through a reviewed yearly import. Do not add the table to realtime; clients can cache by year and version.

```text
  ├─ 1:1 user_settings
  ├─ 1:N dashboard_widgets
  ├─ 1:N dashboard_layouts
  ├─ 1:N connected_accounts
  │     ├─ 1:N integration_permissions
  │     ├─ 1:N integration_cache
  │     └─ 1:N sync_jobs
  ├─ 1:N notifications
  ├─ 1:N projects
  │     ├─ 1:N tasks
  │     ├─ 1:N notes
  │     └─ 1:N calendar_events
  ├─ 1:N tasks
  ├─ 1:N notes
  └─ 1:N calendar_events

```

## 9. RLS policy pattern

For a direct user-owned table:

```text
SELECT/INSERT/UPDATE/DELETE allowed only when auth.uid() = user_id
INSERT additionally forces user_id = auth.uid()
```

For child records:

```text
auth.uid() = child.user_id
AND referenced parent.user_id = auth.uid()
```

For server-managed integration tables:

```text
owner SELECT of safe metadata
no direct browser INSERT/UPDATE/DELETE
service/Edge Function performs mutations after JWT and ownership validation
```

RLS is necessary but not sufficient. Edge Functions must verify the authenticated user, account ownership, input schema, requested action, and allowed scope before using service credentials.

## 10. Retention and deletion

| Data                           | Suggested initial retention                            |
| ------------------------------ | ------------------------------------------------------ |
| Personal records               | Until user deletes; define recovery policy later.      |
| Dashboard layouts              | Current row plus limited audit history only if needed. |
| Gmail cached bodies            | Avoid by default; otherwise shortest practical TTL.    |
| Gmail list metadata            | Minutes to hours; purge on disconnect.                 |
| Search Console aggregates      | 30–90 days, refresh finalized windows less often.      |
| YouTube search cache           | Minutes; no need for long history.                     |
| Sync job success logs          | 14–30 days.                                            |
| Sync job failures/audit events | 30–90 days, sanitized.                                 |
| Khmer holidays                 | Permanent, versioned, supersede rather than overwrite. |

Account deletion must cascade user-owned records, revoke connected accounts where possible, remove token material, cancel sync jobs, and purge provider caches.

## 11. Migration order

1. `profiles`, `user_settings`
2. `dashboard_widgets`, `dashboard_layouts`
3. `projects`, `tasks`, `notes`, `calendar_events`, `notifications`
4. `khmer_holidays` plus controlled import procedure
5. `connected_accounts`, `integration_permissions`, private token storage
6. `integration_cache`, `sync_jobs`

Each migration is complete only after constraints, indexes, RLS policies, policy tests, and generated TypeScript database types are updated.
